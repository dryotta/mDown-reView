## ADDED Requirements

### Requirement: Add a block comment
The application SHALL allow the user to attach a review comment to a specific rendered block (paragraph, heading, code block, blockquote, or list) in an open document by clicking a comment affordance (e.g., a `+` icon in the left margin) that appears on hover next to each block. The comment SHALL be anchored by content hash (`blockHash`) with the nearest preceding heading slug as secondary context (`headingContext`), and the creation-time line number (`fallbackLine`) stored for display only.

#### Scenario: Hover reveals comment affordance
- **WHEN** the user hovers the mouse over a rendered block in the document viewer
- **THEN** a comment icon appears in the left margin of that block

#### Scenario: Open comment input
- **WHEN** the user clicks the comment icon on a block
- **THEN** a comment input area appears inline below that block, ready for text entry

#### Scenario: Submit comment
- **WHEN** the user types text into the comment input and submits it (via a "Save" button or `Ctrl+Enter`)
- **THEN** the comment is saved, the input closes, and a comment indicator is shown in the margin for that block

#### Scenario: Cancel comment
- **WHEN** the user presses Escape or clicks "Cancel" in the comment input
- **THEN** the input closes without saving and no comment is created

### Requirement: Comment anchor stability
When a reviewed document is regenerated by an AI tool and its blocks are reordered or moved, comments SHALL re-attach to matching blocks by content hash. A comment is orphaned only when its `blockHash` no longer exists anywhere in the document.

#### Scenario: Comment survives block reorder
- **WHEN** a document is regenerated with the same paragraph content at a different position
- **THEN** the existing comment re-attaches to the matching paragraph and displays correctly at its new position

#### Scenario: Orphaned comment flagged in panel
- **WHEN** a document is opened and a saved comment's `blockHash` no longer matches any block in the document
- **THEN** the comment is shown in the comments panel with an "orphaned" visual indicator and its original `fallbackLine` for context; no crash or data loss occurs

### Requirement: View existing comments
The application SHALL display comment indicators in the document margin for all blocks that have associated unresolved comments. Clicking a comment indicator SHALL reveal the comment content inline.

#### Scenario: Margin indicator for commented blocks
- **WHEN** a document is opened that has saved comments
- **THEN** comment indicators (e.g., icons or count badges) appear in the left margin at the appropriate blocks

#### Scenario: Expand comment inline
- **WHEN** the user clicks a comment indicator in the margin
- **THEN** the comment(s) for that block expand inline below the block, showing the comment text and timestamp

#### Scenario: Collapse comment
- **WHEN** the user clicks the expanded comment block
- **THEN** the comment collapses back to the margin indicator

### Requirement: Edit and delete comments
The application SHALL allow the user to edit or delete an existing comment. Editing SHALL replace the comment text in-place. Deletion SHALL remove the comment and its margin indicator.

#### Scenario: Edit comment
- **WHEN** the user clicks "Edit" on an expanded comment
- **THEN** the comment text becomes editable, and saving replaces the old text with the new text

#### Scenario: Delete comment
- **WHEN** the user clicks "Delete" on an expanded comment
- **THEN** the comment is removed and the margin indicator disappears (if no other comments remain on that block)

### Requirement: Mark comment as resolved
The application SHALL allow the user to mark a comment as resolved. Resolved comments SHALL be visually distinguished (dimmed with a strikethrough header) and hidden from the comments panel by default. A "Show resolved" toggle in the panel SHALL reveal them.

#### Scenario: Resolve a comment
- **WHEN** the user clicks "Resolve" on an expanded comment
- **THEN** the comment is marked as resolved, its margin indicator changes to a muted resolved style, and it is removed from the default panel view; the tab badge count decreases accordingly

#### Scenario: Show resolved comments
- **WHEN** the user toggles "Show resolved" in the comments panel
- **THEN** resolved comments appear in the panel with a distinct dimmed styling below unresolved comments

#### Scenario: Unresolve a comment
- **WHEN** the user clicks "Unresolve" on a resolved comment
- **THEN** the comment returns to unresolved state and appears normally in the panel; the tab badge count increases accordingly

### Requirement: Comments panel
The application SHALL provide a right-side comments panel that lists all unresolved comments for the currently active document in block order. Clicking a comment in the panel SHALL scroll the document to the corresponding block and expand the comment inline.

#### Scenario: Panel lists unresolved comments
- **WHEN** the comments panel is open and a document with comments is active
- **THEN** all unresolved comments for that document are listed in the panel in block order, showing a preview of the text and timestamp

#### Scenario: Click comment in panel scrolls to block
- **WHEN** the user clicks a comment entry in the comments panel
- **THEN** the document scrolls to bring the commented block into view and the comment expands inline

#### Scenario: Empty state
- **WHEN** the active document has no unresolved comments
- **THEN** the comments panel shows a message: "No comments yet"

### Requirement: Toggle comments panel
The application SHALL allow the user to show or hide the comments panel via a toolbar button or keyboard shortcut (`Ctrl+Shift+C` / `Cmd+Shift+C`).

#### Scenario: Hide comments panel
- **WHEN** the comments panel is visible and the user presses the toggle
- **THEN** the panel hides and the viewer expands to fill the space

#### Scenario: Show comments panel
- **WHEN** the comments panel is hidden and the user presses the toggle
- **THEN** the panel reappears

### Requirement: Comment persistence
The application SHALL persist comments to a sidecar file (`<filename>.review.json`) in the same directory as the reviewed document. The sidecar SHALL use a versioned JSON envelope: `{ "version": 1, "comments": [...] }`. Each comment object SHALL include: `id` (UUID), `blockHash` (8-char hex hash of block raw text), `headingContext` (slug of nearest preceding heading, or null), `fallbackLine` (integer), `text` (string), `createdAt` (ISO timestamp), `resolved` (boolean). Comments SHALL survive application restarts.

#### Scenario: Comments saved with versioned envelope
- **WHEN** the user saves a comment
- **THEN** a `<filename>.review.json` file is created (or updated) containing `{ "version": 1, "comments": [...] }`

#### Scenario: Comments loaded on open
- **WHEN** the user opens a file that has an associated `.review.json` sidecar file
- **THEN** the saved comments are loaded and displayed in the margin and comments panel

#### Scenario: Legacy sidecar migration
- **WHEN** a sidecar file without a `version` field is encountered
- **THEN** the application reads it as a legacy format and migrates it to the current versioned format on the next save, preserving all existing comment data

#### Scenario: No sidecar when no comments
- **WHEN** a document is opened but no comments are added
- **THEN** no `.review.json` file is created

### Requirement: Comment count badge on tab
The application SHALL display a badge on the document tab showing the number of unresolved comments for that document when unresolved comments are present.

#### Scenario: Tab badge shows unresolved count
- **WHEN** a document has one or more unresolved comments
- **THEN** the tab label shows a numeric badge with the unresolved comment count

#### Scenario: Badge disappears when all comments resolved or deleted
- **WHEN** all comments on a document are either deleted or marked as resolved
- **THEN** the badge is removed from the tab label
